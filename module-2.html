What is React?

  • A declarative UI library.
  • Runs on multiple targets: DOM, canvas, native, terminal, etc.
  • Uses a virtual DOM unidirectional data-binding system to only update the parts of the DOM that need to change.
  • A way to bring functional programming to building user interfaces.

Props and Unidirectional Data Flow

In React, data flows in only one direction: from parent to child. This makes it easy to track where bugs are coming from since we have a guarentee that data can only come from one place: a component's parent.

Managing State

In order to do anything useful, we need to track more than just static data. We need some way to track state changes and update the UI accordingly. Again, React shines in this respect. By making your state explicit, React makes it obvious what your state is at any given time. By far the biggest challenge in software is determining the present state of our programs. React helps alleviate this problem in the UI.

Props vs State

So far you've learned about props and state. How do you know when to use state or props? State is for data that you know will change due to some external input. Props can be thought of as static "properties".

To seperate concerns, aim to create small stateless child components that act like functions in that they take a set of arguments (props) and always returns the same value based on that input. Keep shared mutable state in a parent component that also contains the business logic.

React is Functional

As mentioned earlier, you should aim to create small stateless components that act like functions. In fact, React embraces this idea even further by allowing you to use functions instead of classes to create components. They are simply called "stateless functional components".

See the Pen React is Functional by Freddy Rangel (@freddyrangel) on CodePen.

As a general rule, stick to stateless functional components and avoid using state as much as possible. If and only if you need to use state, then you reach for a React class component.
